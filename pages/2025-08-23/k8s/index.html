1. 🎯 Kubernetes 개요

1.1 Kubernetes란?

Kubernetes(k8s)는 컨테이너화된 애플리케이션의 배포, 확장 및 관리를 자동화하는 오픈소스 플랫폼입니다.

🔧 핵심 기능

⚡ 자동화된 배포 및 롤백

선언적 구성을 통한 애플리케이션 배포

🔍 서비스 검색 및 로드 밸런싱

DNS 이름 또는 IP 주소를 통한 컨테이너 노출

💾 스토리지 오케스트레이션

로컬, 클라우드 스토리지 시스템 자동 마운트

🔄 자가 치유

실패한 컨테이너 재시작, 교체, 종료

🔐 시크릿 및 구성 관리

민감한 정보를 안전하게 저장 및 관리

📊 자동 스케일링

CPU 사용률 기반 Pod 수 자동 조정

1.2 Kubernetes의 이점

📈

확장성

수평 및 수직 스케일링 지원으로 트래픽 증가에 유연하게 대응

⚡

가용성

장애 복구 및 무중단 서비스를 통한 높은 가용성 보장

🔄

이식성

온프레미스, 클라우드, 하이브리드 환경에서 동일하게 실행

⚙️

관리성

선언적 구성 및 자동화를 통한 운영 복잡성 감소

💰

효율성

리소스 최적화 및 비용 절감을 통한 경제적 운영

2. 🧩 기본 개념

2.1 핵심 오브젝트

📦 Pod

Kubernetes의 최소 배포 단위로, 하나 이상의 컨테이너를 포함합니다.

특징:
  • 동일한 네트워크 및 스토리지 공유
  • 함께 스케줄링되고 종료됨
  • 고유한 IP 주소 할당
Pod YAML 예제:
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.21
    ports:
    - containerPort: 80

🔄 ReplicaSet

지정된 수의 Pod 복제본이 항상 실행되도록 보장합니다.

ReplicaSet YAML 예제:
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: nginx-replicaset
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21

🚀 Deployment

ReplicaSet을 관리하며 선언적 업데이트를 제공합니다.

Deployment YAML 예제:
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 256Mi

🌐 Service

Pod 집합에 대한 안정적인 네트워크 엔드포인트를 제공합니다.

Service 타입:
ClusterIP

클러스터 내부에서만 접근 가능 (기본값)

NodePort

각 노드의 특정 포트를 통해 외부 접근 가능

LoadBalancer

클라우드 로드 밸런서 생성

ExternalName

DNS CNAME 레코드 생성

Service YAML 예제:
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP
  ports:
  - port: 80
    targetPort: 80
    protocol: TCP
  selector:
    app: nginx

⚙️ ConfigMap

비기밀 구성 데이터 저장

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database.url: "mongodb://localhost:27017"
  log.level: "info"

🔐 Secret

기밀 데이터 저장 (base64 인코딩)

apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  username: YWRtaW4=  # admin
  password: cGFzc3dvcmQ=  # password

🏷️ Namespace

클러스터 내에서 리소스를 논리적으로 분리합니다.

apiVersion: v1
kind: Namespace
metadata:
  name: development

3. ⚙️ Kubernetes 클러스터 설정

3.1 클러스터 설정 방법

🏠 Minikube (로컬 개발용)

로컬 개발 및 테스트를 위한 단일 노드 클러스터

설치 및 시작:
# Minikube 설치 (macOS)
brew install minikube

# 클러스터 시작
minikube start --driver=docker --memory=4096 --cpus=2

# 클러스터 상태 확인
minikube status

# 대시보드 실행
minikube dashboard

🏢 kubeadm (온프레미스)

프로덕션 환경을 위한 멀티 노드 클러스터

마스터 노드 초기화:
# kubeadm 초기화
sudo kubeadm init --pod-network-cidr=192.168.0.0/16

# kubectl 설정
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config

# 네트워크 플러그인 설치 (Calico)
kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml
워커 노드 추가:
# 마스터 노드에서 join 토큰 생성
kubeadm token create --print-join-command

# 워커 노드에서 실행
sudo kubeadm join <master-ip>:6443 --token <token> --discovery-token-ca-cert-hash sha256:<hash>

☁️ 관리형 서비스 (AWS EKS)

클라우드 제공업체의 관리형 Kubernetes 서비스

AWS EKS 클러스터 생성:
# eksctl 설치
curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin

# EKS 클러스터 생성
eksctl create cluster \
  --name my-cluster \
  --version 1.21 \
  --region us-west-2 \
  --nodegroup-name linux-nodes \
  --node-type m5.large \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 4

3.2 kubectl 설정 및 사용법

🔧 기본 명령어

클러스터 정보
# 클러스터 정보 확인
kubectl cluster-info

# 노드 목록 조회
kubectl get nodes
Pod 관리
# Pod 목록 조회
kubectl get pods
kubectl get pods -o wide
kubectl get pods -n kube-system
상세 정보
# 리소스 상세 정보
kubectl describe pod <pod-name>

# 로그 확인
kubectl logs <pod-name>
kubectl logs -f <pod-name>  # 실시간 로그
컨테이너 접속
# 컨테이너 접속
kubectl exec -it <pod-name> -- /bin/bash

# 포트 포워딩
kubectl port-forward <pod-name> 8080:80

📄 YAML 파일 관리

# 리소스 생성
kubectl apply -f deployment.yaml

# 리소스 업데이트
kubectl apply -f deployment.yaml

# 리소스 삭제
kubectl delete -f deployment.yaml

# dry-run으로 확인
kubectl apply -f deployment.yaml --dry-run=client

# 리소스를 YAML로 출력
kubectl get deployment nginx-deployment -o yaml

4. 📋 워크로드 관리

4.1 Deployment 전략

🔄 Rolling Update

기본 배포 전략으로, 새 버전을 점진적으로 배포합니다.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-deployment
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 2  # 최대 2개 Pod까지 사용 불가
      maxSurge: 2        # 최대 2개 Pod까지 추가 생성
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: myapp:v2
        readinessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10

🔵🟢 Blue-Green Deployment

두 개의 완전한 환경을 사용하여 무중단 배포를 수행합니다.

# Blue 환경 (현재 프로덕션)
kubectl apply -f blue-deployment.yaml

# Green 환경 (새 버전) 배포
kubectl apply -f green-deployment.yaml

# 트래픽 검증 후 서비스 전환
kubectl patch service myapp-service -p '{"spec":{"selector":{"version":"green"}}}'

# Blue 환경 정리
kubectl delete -f blue-deployment.yaml

🐦 Canary Deployment

트래픽의 일부만 새 버전으로 라우팅하여 점진적으로 배포합니다.

# Canary 배포용 Deployment (트래픽의 10%)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-canary
spec:
  replicas: 1  # 전체 10개 중 1개
  selector:
    matchLabels:
      app: myapp
      version: canary
  template:
    metadata:
      labels:
        app: myapp
        version: canary
    spec:
      containers:
      - name: app
        image: myapp:v2
